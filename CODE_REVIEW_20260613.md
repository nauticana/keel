# Keel Code Review - 2026-06-13

Scope: reviewed `README.md` and the Go library code from completeness, correctness, security, and accuracy perspectives. Keel is treated as a shared downstream dependency, so findings emphasize generic framework behavior and cross-project blast radius.

## Additional Notes

- **Fixed.** `LocalUserService.GenerateOTP` ignored the `crypto/rand.Int` error before calling `code.Int64()`. Now returns the error loudly.
- **Fixed.** `AbstractHandler.ParseSession` only accepted uppercase `Bearer `, while `HttpBackend.SSOMiddleware` accepted it case-insensitively. `ParseSession` now uses the same case-insensitive (`EqualFold`) check + `TrimSpace`, so `RequireSession` behaves identically inside and outside the backend middleware (RFC 7235).
- **By design — caveat documented.** `CreateCheckout` intentionally does not overwrite caller-supplied `metadata["user_id"]` on authenticated checkout, to support admin/impersonation flows. The override is not privilege-gated in the shared handler (downstream-specific roles can't be evaluated here), so changing the default would risk breaking legitimate impersonation flows. Left as-is; added a call-site security caveat directing downstreams that expose checkout to untrusted users to reject/admin-gate caller-supplied `metadata["user_id"]`.

## Resolutions (applied 2026-06-13)

All changes are backward-compatible for downstream consumers of the default implementations. The previously-red `pgsql` suite is now green; `go test ./...` passes across all packages.

- **KR-001** — Session scope is now non-overridable on generic CRUD. `pgsql.TableServicePgsql.Get`/`Delete`: PartnerSpecific reads pin `partner_id` to the session partner unless the caller holds a `data.GlobalRoleIDs` role (preserves the deliberate SUPER/BUSINESS_ADMIN cross-partner capability via the existing `IsGlobalRole` gate — no *new* access is granted, only the normal-user leak closed); UserSpecific reads force `user_id = caller` when owner-scoped, and Delete forces it unconditionally. The relation-cascade path is untouched (it derives child filters from an already-authorized parent, not from caller input), so nested REST keeps working. New regression tests cover foreign-partner coercion, global-role preservation, owner-lock force, and the `bypass_scope` admin path.
- **KR-002** — Added an optional `payment.WebhookReclaimer` interface; `WebhookProcessor.Process` now atomically re-claims a `StatusFailed` row and re-runs the handler on provider retry (single-flight via a conditional `UPDATE … WHERE processing_status='F' RETURNING id`). The default `SQLWebhookRepository` implements it; custom repositories that don't keep the old conservative behavior (no compile break). Test double aligned to real SQL `Exists` semantics; added a retry-after-failure test; concurrency/charge-once test still passes under `-race`.
- **KR-003** — `CheckPermission` / `CheckActionPermission` now match the actual `QCheckAuthorization` projection: exact `low_limit` or `'*'` only. Removed the dead glob/range branches (the SQL filter never surfaced those rows, so this is behavior-preserving and fails safe). Documented exact + `'*'` as the contract; did **not** widen the SQL, since enabling glob/range would *expand* access on upgrade — a dangerous direction for a shared security library.
- **KR-004** — pgsql test helpers now return the three-column `[low_limit, high_limit, bypass_scope]` row that `CheckPermission` reads at `rec[2]`. Test-only fix.
- **KR-005** — `qGenerateOTP` binds `expires_at` from `--otp_ttl_seconds` (was hard-coded `INTERVAL '2 minutes'`), so the DB row lifetime matches the value echoed to the client and `OTPHandler.OTPTokenTTL`. Fixed the stale "2-minute" comment on `OTPTokenTTL`.
- **KR-006 (deferred)** — Valid, but a low-risk P2: the response is already hard-capped at `MaxListPageSize` (1000) and the full-scan limitation is documented in code. A real fix requires widening `rest.RestAPI` / `data.TableService.Get` to thread `LIMIT`/`OFFSET` (+ deterministic ordering) through the handler → RelationAPI → TableService chain — a breaking signature change across an interface multiple downstream projects implement. Deferred to a coordinated change (could be done backward-compatibly via an optional `Pager` extension interface) rather than break downstream now.

## Positive Observations

- The README is unusually explicit about migration contracts and security rationale, which made mismatches visible.
- Several historically risky areas have good defenses: webhook signature verification before logging, CORS origin allowlisting with `Vary: Origin`, JWT alg pinning, strict helpers for sensitive JSON endpoints, constant-time OTP comparison, and owner-pinning on UserSpecific insert/update.
- Most DB access is outside handlers, with handlers acting as HTTP bridges into services or framework abstractions.
